# Copilot Instructions for GAMA Project

This file provides instructions to GitHub Copilot to ensure consistency and quality in the development of the GAMA project.

## SOLID Principles
When writing or modifying code, please adhere to the SOLID principles of object-oriented design:
- **Single Responsibility Principle (SRP):** A class should have only one reason to change.
- **Open/Closed Principle (OCP):** Software entities (classes, modules, functions) should be open for extension, but closed for modification.
- **Liskov Substitution Principle (LSP):** Subtypes must be substitutable for their base types.
- **Interface Segregation Principle (ISP):** No client should be forced to depend on methods it does not use.
- **Dependency Inversion Principle (DIP):** High-level modules should not depend on low-level modules. Both should depend on abstractions.

## Unit Testing
- For any new code (e.g., new features, bug fixes), create corresponding unit tests.
- Where applicable, follow the Test-Driven Development (TDD) approach.
- Ensure that tests cover both success scenarios and edge cases.
- All existing and new tests must pass before code is considered complete.

## Documentation
- When you introduce changes, make sure to update the relevant documentation files.
- Key files to keep updated are:
    - `README.md`: For general project information.
    - `QUICK_START.md`: For user and developer quick start guides.
    - `DOCUMENTATION_INDEX.md`: If you add or remove documentation files.
- Add comments to the code where necessary to clarify complex logic.

## Language
- All code, including comments, variable names, and documentation, must be written in English.
- This is to ensure the project is accessible to a wider audience of developers, even though the primary user base may be Portuguese-speaking.

## PowerShell Scripting
- **Character Encoding:** Ensure all PowerShell (`.ps1`) files are saved with `UTF-8 with BOM` encoding to prevent issues with special characters.
- **Avoid Special Characters in Output:** Do not use Unicode symbols or special characters (e.g., ✓, ✗, →) in `Write-Host` or other output commands. Use plain ASCII text like "OK", "ERROR", "WARNING", "Success", "Failure" instead. This prevents rendering issues in different terminal environments.
- **Quoting and Arguments:** Be extremely careful with quotes when calling external commands (like `jpackage` or `mvn`). To avoid parsing errors, store complex arguments, especially those containing variables or special characters (like `$APPDIR`), in a separate PowerShell variable before passing them to the command. For example:
  ```powershell
  # Good practice
  $javaOptions = '--enable-preview -Djava.library.path=$APPDIR'
  & jpackage --java-options $javaOptions
  
  # Avoid
  & jpackage --java-options '--enable-preview -Djava.library.path=$APPDIR'
  ```

## Installer Scripts (`build-installer.ps1`)
- **Application Version:** When creating a new release, remember to update the `--app-version` parameter in the `jpackage` command.
- **Upgrade UUID:** The `--win-upgrade-uuid` must **remain constant** across all versions. Do not change it. This UUID ensures that new installations correctly replace older versions.
- **File Paths:** Ensure all file paths passed to external commands are correctly quoted and handled to prevent parsing issues.
